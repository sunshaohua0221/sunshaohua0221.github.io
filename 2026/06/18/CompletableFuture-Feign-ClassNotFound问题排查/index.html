CompletableFuture-Feign-ClassNotFound问题排查

最近在一次业务开发中，需要将用户行为记录的逻辑改为异步执行，于是很自然地使用了 CompletableFuture.runAsync() 来包装 Feign 接口调用。本地测试一切正常，但部署到开发环境后，却频繁出现 ClassNotFoundException。

有趣的是，把代码改回同步调用就一切正常了。

更让人困惑的是：生产环境配置是 request: 1、limit: 4 核，没有问题；而 STG 环境配置是 request: 1、limit: 2 核，却无法复现。同一个代码库，在不同环境表现不同——这本身就说明问题跟业务代码无关，而是跟运行时的某些底层机制有关。

2. 根本原因分析

  1. CompletableFuture 默认使用 ForkJoinPool
    CompletableFuture.supplyAsync() 和 runAsync() 如果不指定线程池，默认会使用 ForkJoinPool.commonPool()。

  2. JDK 11 中 ForkJoinPool 的类加载器行为变化
    这是问题的第一个关键点。

在 JDK 8 中，ForkJoinPool 的工作线程会继承父线程的上下文类加载器（Context ClassLoader）。但从 JDK 11 开始，ForkJoinWorkerThread 默认会将线程的上下文类加载器强制设置为系统类加载器（SystemClassLoader / AppClassLoader），而不再继承父线程的类加载器。

这意味着：即使主线程使用的是应用类加载器（AppClassLoader）来加载业务类，一旦任务被提交到 ForkJoinPool.commonPool()，工作线程的类加载器就变成了系统类加载器，无法访问应用自己的类。

  1. Feign 接口是懒加载的
    这是问题的第二个关键点。

Spring Cloud 对 Feign 客户端默认采用懒加载策略——只有在第一次调用时，才会初始化 Feign 客户端、生成动态代理、加载相关的依赖类。

  1. 两个问题叠加，产生 ClassNotFound
    当 CompletableFuture 的异步任务（运行在 ForkJoinPool 线程中）首次调用 Feign 接口时：

  2. Feign 的懒加载机制触发，开始动态加载类、生成代理

  3. 但当前线程（ForkJoinWorkerThread）的上下文类加载器是 SystemClassLoader

  4. SystemClassLoader 找不到应用业务类（这些类在 AppClassLoader 的类路径中）

  5. 于是抛出 ClassNotFoundException

而如果 Feign 接口已经被调用过一次（类已经加载完成），再放到异步任务中执行就不会报错——这就是为什么有些场景下问题时有时无。

  1. 为什么 STG 环境（2 核）无法复现？
    这是排查过程中最迷惑的一点。

ForkJoinPool.commonPool() 的并行度默认等于 Runtime.getRuntime().availableProcessors() - 1。当 CPU 核心数大于等于 3 时，ForkJoinPool 会创建多个工作线程；而当核心数小于 3（比如 2 核）时，ForkJoinPool 的行为可能有所不同。

生产环境是 4 核 → ForkJoinPool 正常工作 → 问题出现
STG 环境是 2 核 → ForkJoinPool 行为异常/降级 → 问题无法复现

这也是为什么在本地（通常是多核）能复现，而在某些低配环境反而没问题——不是环境越强越稳定，恰恰相反。

4. 解决方案

方案一：使用自定义线程池（推荐）
最直接、最干净的解决方案——不要使用 ForkJoinPool.commonPool()，而是传入自定义的 Executor。

使用 ThreadPoolExecutor 创建的自定义线程池，其工作线程会继承父线程的上下文类加载器，因此能够正常加载应用类。

方案二：自定义 ForkJoinPool 并设置类加载器

这样后续的异步调用就不会触发懒加载，也就不会遇到类加载器问题了。

但这只是绕过问题，不是根本解决。如果 Feign 接口较多，预热代码会变得很臃肿。

5. 最佳实践总结

  1. 永远不要在生产代码中直接使用 CompletableFuture.runAsync() 或 supplyAsync() 的无参版本——显式传入自定义线程池是基本素养。
  2. IO 密集型任务（如 Feign 调用）不要用 ForkJoinPool——ForkJoinPool 是为 CPU 密集型任务设计的，IO 密集型任务应该使用 ThreadPoolExecutor 并合理配置线程数。
  3. 理解 JDK 版本差异——从 JDK 9 开始，Java 的模块化和类加载机制发生了不少变化。升级 JDK 版本时，要多留意这些底层行为的变化。
  4. Feign 的懒加载是一个“特性”也是一个“坑”——了解它的工作机制，在涉及多线程/异步场景时提前做好预案。
